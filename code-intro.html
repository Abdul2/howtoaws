What the course covers

Step 1 - Download software

You will need a few things on your laptop when working with GitHub and code. A text editor so you can edit and write code. We recommend Atom which integrates nicely with GitHub. There are lots of others to try.

You'll also need to manage your work and connect to GitHub. To do this we recommend either downloading GitHub Desktop or using command line with the Terminal app on your mac (you need to run this command in terminal to use command line xcode-select --install).

Step 2 - Find repo

A repo is a project and its history in GitHub. You can look through all the Home Office repos and GDS. Our practice repo will be https://github.com/owenm6/mockservice

Step 3 - Clone repo

You need to clone or download a repo to your computer so you can work on it before uploading it back to GitHub. To clone a repo:

Directly from GitHub

Press the big green button that says ‘clone or download’. Make sure you move it to somewhere easy to find on your computer.

GitHub Desktop

Go to file > clone and input the URL of the repo.

Command Line

  1. (Optional) Create a new folder mkdir name_of_folder
  2. Navigate to that folder cd name_of_folder
  3. Clone repo git clone url-of-repo

Step 4 - Branch

Branching is a way of working away from the live product. People create branches for new features or to test something. When they finish the work they request for it to be merged back into the main (master) branch. To create a branch:

GitHub Desktop

Go to branch > new branch and give it a name

Command Line

  1. Navigate to your repo folder with the cd command
  2. Create a branch git branch name
  3. Change to that branch git checkout name

Step 5 - Open project in a text editor

To open a repo in your text editor, drag the folder into your text editor. In command line you can do this really quickly with this funky command atom .

Intro to HTML

HTML is the language we use to create the structure and content of a webpage. The types of things you’ll create in HTML are paragraphs, lists and images. Most of the time in HTML, we use opening and closing tags. So a paragraph would look like <p>I love cats.<p>

Step 6 - Make your changes

To change something in a project, first find the file you need to work with. In the prototype kit, the things you will be interested in can be found in the app folder. In the views folder you can find the HTML pages. In the assets folder is the CSS and JavaScript files.

A good place to start is the index.html file. The index page is usually the homepage or start page of a website.

Step 7 - Commit and push

Once you’ve made changes, save them. In Atom, you will notice the colour of the file changes to yellow. This tells you that you have changes locally that need to be uploaded back to Github.

The first step in uploading changes is to commit those changes. When you commit a change, what you are doing is taking a snapshot of your changes.

To create a commit:

GitHub Desktop

Once you have saved some changes, GitHub desktop should show you exactly what you changed. To commit, add a summary message and description in the box in the bottom left-hand corner and click the button ‘commit to name_of_branch’.

Command line

  1. Navigate to your repo folder with the cd command
  2. Check the status of your project with git status. This shows you what files you have changed.
  3. Do a git add. This tells git that you want to add certain files to a commit. You can add all the files that have changes using git add .. Or you can specify which files you want to add to a commit.
  4. Now commit using git commit -m “your commit message”

The message added to a commit is really important. It needs to be descriptive and easy to understand. The great thing about a commit is you can go back in time and revert to the state of a project at a previous commit. To make this easier, make sure you create a commit message that is descriptive.

To push (upload your changes):

GitHub Desktop

Click the push origin button

Command line

git push

Step 8 - Creating a pull request

A pull request tells the owner of a repo that you have made some changes and you would like them to be go live.

You can create a pull request on GitHub. Once you have pushed, you can go to branch > create pull request in GitHub desktop which take you to GitHub to do it. If you are using command line, the easiest way is probably to create one on the project page on GitHub.

Helpful terminology

Git

Git refers to the version control software that lets you save snapshots of all your changes.

Github

Github is a website where you can host Git projects, collaborate and suggest improvements and browse thousands of open source repos.

Repository

A repository or repo is all of the project files and the history of any changes you have made.

Local

A way of distinguishing the version of a repo on your own computer versus the version of the repo in a central place, for example Github.

Remote

The version of the repo in the central place.

Branch

A branch is a way to isolate changes you want to make. You don’t have to worry about breaking anything because it is another copy of the code - you can always go back to an earlier version.

A branch is a great way to experiment. In a branch you can work freely away from the live product without worrying about breaking anything.

Branches are also useful when you’re working in a team, so that you can keep anything you’re working on completely separate from anyone else.

This is a great guide to branching.

Fork

Forking is a way of making your own copy of an entire repo. It’s useful if you want to later contribute your changes back to the original copy by using a pull request.

Master

The master branch is usually where the code that can be made live is kept . You would normally only make updates to a live website from the master branch. It’s usually good practice to protect the master branch so that changes to it are controlled and checked by another person.

Merge

Merging is a how GitHub combines two branches back together again. Sometimes two people may have made changes to the same part of a file (a conflict). If so, Git allows you to choose which change should be committed.

Commit

A commit change is a snapshot of the changes you have made.

Push

Upload your local repo to a shared one hosted on the web (Github for example)

Pull

Download and synchronise the latest changes to a shared repo.

Pull request

A way of ask the owner of a repo to accept changes you have made.
