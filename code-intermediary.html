What the course covers

In this training, we take participants through using the GOV.UK prototyping kit, and how to get a prototype live using GitHub and Heroku.

Step 1 - Find and download the prototyping kit on Github

You can find the prototype kit here - github.com/alphagov/govuk_prototype_kit. Download it to your computer. Make sure to change it’s name to something memorable (the name of your service or what you are testing). Then put it somewhere on your computer that is easy to find. A lot of people put their prototypes in sites.

You’ll use a copy of the kit for each different prototype you want to make - so this example shows 7 different prototypes.

Step 2 - Intro to command line

When working with the kit, there are some things that can only be achieved using command line. Command line is a way of interact with a computer without the graphical user interface (GUI).

On a Mac, you use command line in an app called Terminal. Open Terminal - a quick way to do this on Mac is to press cmd ⌘ space and start typing terminal.

Step 3 - Install Xcode and node if you haven’t already

Xcode is a selection of developer tools. One of those tools is command line so you need Xcode installed. To install it, in Terminal input xcode-select --install and press enter. It will walk you through installing. If you already have it, it will say the tools are already installed.

You also need to install Node onto your computer. You can do this here - nodejs.org. If you think you already have node, you can check by inputting this command into Terminal node -v

Step 4 - Run the kit

To run the kit, first navigate in terminal to prototype you have already downloaded. There are a few ways to do this. You can drag the folder into terminal application. Or you can navigate using commands. To do this, you will use the cd command. This means change directory. For example if you have put your prototype in your sites folder, you would input this command into terminal cd sites/name_of_prototype_folder.

Once you have navigated to the folder in terminal, you can run the kit with two commands. npm install followed by npm start. npm install is a command you only have to run once to install all the things the kit needs to work. Every other time you want to work on your prototype after that, all you need to do is navigate to the prototype folder and use the command npm start.

If you open your browser and go to this address - http://localhost:3000/ - you will find the kit welcome page.

!! NOTE - if the kit is running, you will need to open a new terminal window if you want to do anything in command line like committing or publishing changes.

Step 5 - Exploring the kit

Now the kit is running, we want to open it in a text editor so we can start making changes. In your text editor go to file > open and find your project.

Once opened, you’ll find most of the files you need in the app folder.

Step 6 - Making a new page

You can make a new page from scratch or use one of the page templates. The page templates can be found in docs > views > examples.

Put your new pages in app > views

Step 7 - Upload your prototype to GitHub

You can do this with Command Line or GitHub Desktop. If you have taken the intro course you will have already linked your local computer to your GitHub account so this step becomes easier.

GitHub Desktop

Select File > add local repo. Navigate to your project and add it. Now commit your changes. Click the publish button in the top right.

Command line

Go to GitHub.com and create a new repo in your account. Then in command line (making sure you have opened up a new window if you are running the kit) navigate to your prototype folder using the cd command.

  1. Initialise - git init
  2. Add files - git add .
  3. Commit - git commit -m “message”
  4. Add remote - git remote add origin url_of_github_repo
  5. Push - git push -u origin master

Step 8 - Creating a heroku app and password protecting it

Once your prototype is in GitHub, you can make it live using a service called Heroku. Create an account with Heroku. Then create a new app. On the app dashboard deployment method, connect to GitHub and choose your prototype repo.

The final step is to add a username and password to protect the prototype. In the settings tab, click the button labeled ‘Reveal Config Vars’. In the ‘key’ field input USERNAME and in the value give a username. Add that and then create another one for PASSWORD.

Helpful terminology

Git

Git refers to the version control software that lets you save snapshots of all your changes.

Github

Github is a website where you can host Git projects, collaborate and suggest improvements and browse thousands of open source repos.

Repository

A repository or repo is all of the project files and the history of any changes you have made.

Local

A way of distinguishing the version of a repo on your own computer versus the version of the repo in a central place, for example Github.

Remote

The version of the repo in the central place.

Branch

A branch is a way to isolate changes you want to make. You don’t have to worry about breaking anything because it is another copy of the code - you can always go back to an earlier version.

A branch is a great way to experiment. In a branch you can work freely away from the live product without worrying about breaking anything.

Branches are also useful when you’re working in a team, so that you can keep anything you’re working on completely separate from anyone else.

This is a great guide to branching.

Fork

Forking is a way of making your own copy of an entire repo. It’s useful if you want to later contribute your changes back to the original copy by using a pull request.

Master

The master branch is usually where the code that can be made live is kept . You would normally only make updates to a live website from the master branch. It’s usually good practice to protect the master branch so that changes to it are controlled and checked by another person.

Merge

Merging is a how GitHub combines two branches back together again. Sometimes two people may have made changes to the same part of a file (a conflict). If so, Git allows you to choose which change should be committed.

Commit

A commit change is a snapshot of the changes you have made.

Push

Upload your local repo to a shared one hosted on the web (Github for example)

Pull

Download and synchronise the latest changes to a shared repo.

Pull request

A way of ask the owner of a repo to accept changes you have made.
